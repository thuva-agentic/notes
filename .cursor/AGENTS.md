## Memory Bank Structure

The memory bank lives inside the `.agent` folder. It consists of core project files and per-ticket working files, all in markdown format. Files build upon each other in a clear hierarchy:

```
flowchart TD
  PB[projectbrief.md] --> PC[productContext.md]
  PB --> SP[systemPatterns.md]
  PB --> TC[techContext.md]

  PC --> AC[activeContext.md]
  SP --> AC
  TC --> AC

  AC --> TK[tickets/active/DM-XXXX/spec.md]
  TK --> TA[tickets/active/DM-XXXX/tasks.md]
```

### Core Files (Required)

1. `projectbrief.md`
   - Foundation document that shapes all other files

3. `systemPatterns.md`

5. `activeContext.md`
   - Pointer to the currently active ticket and its files

   - Active architectural considerations
   - Does NOT contain task lists â€” those live in the ticket folder

# Style

I write code in a functional-first style, but balance purity with practicality.

- I break problems into small, reusable, composable functions.
- I use immutable data structures and pure functions for core logic.
- I isolate side effects at the edges of the system (IO, logging, network calls).
- I favor functional patterns (`map`, `filter`, `reduce`) over imperative loops.
- I use type annotations consistently.
- I choose clarity over cleverness.

I apply this style in Python, TypeScript, Scala, and Kotlin. In other languages I follow the idiomatic conventions of that language while keeping the same principles.

## REST / HTTP API (Remote Backend)

Use the project's HTTP layer for **server-side** data (CRUD, auth, uploads to backend).

### Rules

1. **Do not** call `fetch` / `axios` directly in presentational components. Use the project's hooks or service modules.
2. Use the established API client (from a provider, singleton, or factory) so auth headers, base URL, and error handling stay centralized.
3. Organize by **domain** (one folder or module per resource): types, query keys, read hooks, mutation hooks, validation schemas.
4. Follow existing patterns for:
   - **Reads** â€” cached queries (e.g. `useQuery`, custom `useCreateQuery`)
   - **Writes** â€” mutations with cache invalidation or optimistic updates
   - **Query keys** â€” stable enums or constants, not ad-hoc strings
5. Colocate **Zod** (or project validator) schemas with the domain when forms or payloads are validated.
6. Dynamic URL segments: follow how the project substitutes `params` (e.g. `/resource/${id}`).
7. Copy the **nearest similar domain module** as a template for new resources.

### Auth

- Tokens and session lifecycle belong in auth store / interceptors â€” do not reimplement 401 handling in every hook.
- Never log or commit secrets, API keys, or tokens.

## Electron IPC (Preload Bridge)

Use IPC for **OS-level or main-process-only** work, for example:

- Local filesystem read/write
- Native dialogs, notifications, shell open
- Secure credential storage
- Auto-update, window controls
- Hardware or OS integrations unavailable in the renderer

### Security rules (required)

1. **`contextIsolation: true`** â€” renderer must not import `electron` or touch `ipcRenderer` directly.
2. Expose a **minimal, typed API** via `contextBridge.exposeInMainWorld` in preload.
3. **Allowlist** channels in preload (`invoke` / `on` / `send`) â€” reject unknown channel names.
4. Adding IPC requires **all** of:
   - Handler in main (`ipcMain.handle` / `ipcMain.on`)
   - Allowlist entry in preload
   - Exposed method on the bridge object
   - TypeScript types on `Window` (or shared types package)
5. Validate and normalize paths in main before FS access; prevent directory traversal.
6. Optional chaining in renderer when the bridge may be absent (tests, web fallback): `window.electronAPI?.method?.()`.

### When to use HTTP vs IPC

| Need | Layer |
|------|--------|
| Remote server CRUD, auth, cloud storage | HTTP + project API hooks |
| Local files, OS integration, secure storage | IPC / preload bridge |
| Pure UI state | Component state or client store |

Put reusable renderer-side IPC wrappers in `src/services/` (or equivalent), not duplicated in every component.

### Styling rules
